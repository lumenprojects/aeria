# Aeria — Гайдлайн проекта

Этот документ описывает, что именно сделано в текущем фундаменте Aeria и на что он рассчитан. Цель — держать структуру предсказуемой и не допускать расползания в случайные локальные решения.

## Назначение
Aeria — это SPA-сайт для ранобэ-проекта: эпизоды (главы), персонажи, атлас (весь остальной лор). Контент хранится в Markdown, импортируется в PostgreSQL и индексируется в Typesense для глобального поиска.
- Язык интерфейса и основного контента: русский.
- Переводный слой не внедряется в v1 (кроме `title_native` у главы как отдельного поля данных).

## Архитектура
- **Monorepo**: единый репозиторий, чтобы не дублировать типы/контракты и держать единый стиль.
- **Web**: React + Vite + TypeScript + React Router + Framer Motion.
- **API**: Fastify + TypeScript + Zod + pg.
- **DB**: PostgreSQL, миграции через `node-pg-migrate`.
- **Search**: Typesense (через очередь `search_queue`).

## Структура репозитория
- `apps/web` — фронтенд SPA.
- `apps/api` — backend API.
- `packages/shared` — общие типы и Zod-схемы DTO.
- `packages/db` — миграции Postgres.
- `packages/content` — парсер Markdown, валидатор, импортер, очередь индексации.
- `content/` — исходные Markdown файлы.
- `assets/` — ассеты проекта (images/flags/textures/icons/fonts).
- `docs/` — UX/UI правила и системная документация.

## Контент и импорт
Контент хранится в `content/*` как Markdown с YAML frontmatter.

Ключевые правила импорта:
- **Idempotent re-import**: у каждой записи `source_path` и `content_hash`.
- **Удаление**: при исчезновении файла запись архивируется через `archived_at`.
- **Транзакции**: импорт идёт батчами, с rollback при ошибке.
- **Dry-run**: есть режим `--dry-run` с diff-отчётом.
- **Проверка ссылок**: все slug-ссылки валидируются до записи.
- **Markdown sanitation**: входной Markdown нормализуется, HTML вычищается.
- **Аватары сущностей**: `characters.avatar_asset_path` обязателен; для `locations` и `atlas_entries` поле опционально.
- **Пути ассетов**: `avatar_asset_path` принимается только в формате абсолютного web-path внутри проекта (`/assets/...`).
- **reading_minutes**: единая формула 180 слов/мин, округление вверх.
- **schema_version**: версия формата контента фиксируется в `content/schema.json`.
- **Логи импорта**: `import_runs`, `import_errors`.
- **Поиск**: Typesense обновляется **не** в транзакции, а через очередь и worker.
- **Порядок загрузки**: `countries → locations → series → episodes → characters → atlas → links`.

## База данных
Ключевые сущности:
- `episodes`, `episode_series`, `characters`, `atlas_entries`, `countries`, `locations`.
- M:N: `episode_characters`, `episode_locations`.
- 1:N: `character_quirks`, `character_rumors`.
- Связи атласа: `atlas_links` (направленные связи между сущностями).

Уникальности:
- `episodes(global_order)`
- `episodes(series_id, episode_number)`
- join-таблицы и `atlas_links` с уникальными индексами.

## API
Эндпоинты:
- `/api/home`
- `/api/episodes`, `/api/episodes/:slug`
- `/api/series`, `/api/series/:slug`
- `/api/characters`, `/api/characters/:slug`
- `/api/atlas`, `/api/atlas/:slug`
- `/api/countries`, `/api/locations`
- `/api/search`
- Фильтрация на list-эндпоинтах: `page`, `limit`, плюс сущностные фильтры (`series`, `country`, `kind`).
- `archived_at` исключается из выдачи.
- Каждый публичный entity DTO содержит канонический `url`, чтобы фронт не собирал маршруты вручную.
- В response добавлены data для флагов стран:
  - `/api/episodes` item содержит `country` (для рендера `Flag`);
  - `/api/characters/:slug` содержит `country`;
  - `/api/atlas/:slug` содержит `country`.
- `/api/characters` исключает `listed = false`, но detail `/api/characters/:slug` остаётся доступен по прямому slug.
- `/api/home` отдаёт snapshot для главной:
  - `latest_episode` с `series`, `participants`, `country`;
  - `about_profile` из скрытого персонажа с `home_featured = true`;
  - `world_quote` c цитатой формата `quote/source` для блока "Услышано".

## Поиск
- Typesense индексирует эпизоды, персонажей, атлас, серии, страны, локации.
- Глобальный поиск доступен в навбаре (`Ctrl/Cmd + K`).
- Индексация идёт через очередь `search_queue` и воркер.
- Персонажи с `listed = false` не попадают в публичный список и в поисковый индекс.

## Frontend и навбар
- SPA без перезагрузок.
- `/` — продуктовая главная, собираемая из `/api/home`.
- `/showcase` — внутренняя витрина ассетов/компонентов, не пользовательская главная.
- Глобальный навбар:
  - режим разделов
  - режим breadcrumbs внутри карточек
  - режим чтения главы (полоса прогресса)
- Поиск и системные действия всегда доступны.
- На mobile — нижняя панель навигации.
- Глобальный поиск запускается `Ctrl/Cmd + K`, результаты группируются по типам.

### Navbar v1 (зафиксировано)
- Навбар `sticky`, фон на всю ширину экрана, без верхних/боковых отступов.
- Контент навбара ограничен общей шириной сайта `width-wide`, а не full-width.
- Внутренний vertical inset верхнего nav: `--space-md` сверху и снизу от контента.
- Левый блок:
  - на корневых страницах: `Главная / Эпизоды / Персонажи / Атлас`;
  - на детальных страницах: `Назад` + breadcrumbs.
- Расстояние между первичными разделами nav: `--space-md`.
- Компактные расстояния внутри связанных control-групп (`Поиск`, счётчик главы, `День/Ночь`, `Aa`, `...`, detail breadcrumbs): `--space-sm`.
- Центр: брендовый SVG-логотип `Aeria`.
- Правый блок:
  - `Поиск`
  - индикатор главы `current/total` (только на странице чтения эпизода)
  - `День/Ночь`
  - `Aa`
  - `...`
- Индикатор главы в nav не кликабелен, это только контекст.
- Поиск открывается **inline** под навбаром (не модалка).
- Inline-search использует тот же компактный ритм `--space-sm` для inset и внутренних зазоров.
- В пустом запросе поиск показывает недавние открытые записи (recent).
- Полоса прогресса чтения показывается в верхней границе nav только в режиме чтения эпизода.
- На mobile сохраняется верхний nav и дублируется нижняя панель с разделами.
- Нижний mobile-nav использует `--space-md` как vertical inset.

### Navbar: подменю
- Панели `Aa` и `...` используют только системные отступы:
  - inset панели: `--space-sm`;
  - расстояние между полями панели: `--space-md`;
  - расстояние между label и control внутри поля: `--space-sm`.
- Типографика `Aa` и `...` живёт на том же UI token, что и остальной интерфейс; локальные размеры шрифта в rem для этих панелей запрещены.

### Navbar: сохранение настроек
- Источник истины для настроек навбара: cookie `aeria-theme`.
- В cookie хранятся:
  - `theme`, `mode`
  - `fontUi`, `fontHeading`, `fontBody`
  - `noise`, `tapEffect`
- Параметры cookie:
  - `path=/`
  - `max-age=31536000` (1 год)
  - `sameSite=Lax`
  - `secure` при `https`
- Миграция: если cookie отсутствует, старые настройки из `localStorage` читаются один раз и переносятся в cookie.

### Navbar: поведение меню
- Меню `Aa`:

  - закреплённые наборы шрифтов:
    - UI: `IBM Plex Sans`, `Manrope`, `Golos Text`
    - Headings: `Playfair Display`, `Cormorant Garamond`
    - Text: `Lora`, `Noto Serif`
- Меню `...`:
  - `Style`: `Paper`, `Stone`, `Coral`, `Amoled`
  - `Noise`: `On/Off`
  - `Tap Effect`: `None/Ripple` (на текущем этапе только хранение настройки).

## Дизайн-система
- Три типографические роли: `ui`, `heading`, `body`.
- Строгая иерархия отступов и шрифтов. Используется ограниченный общий список размерностей и не создаются никакие новые ситуативные решения.
- Ширина колонок основного сайта строго трёх видов `narrow`, `medium`, `wide`.
- В коде существуют только три layout-утилиты ширины: `width-narrow`, `width-medium`, `width-wide`.
- Локальные исключения вида `home-*`, `page-*`, `content-*` для ширины запрещены.
- Темы: `Paper`, `Stone`, `Coral`, `Amoled` (каждая light/dark).
- Панель Aa: отдельные шрифты для `UI / Headings / Text`, настройки сохраняются в cookies.
- Компоненты на базе **Radix UI** + кастомные headless при отсутствии примитивов:
  - `Accordion`, `Aspect Ratio`, `Avatar`, `Badge`, `Breadcrumb`, `Button`, `Button Group`, `Carousel`, `Checkbox`, `Combobox`, `Input`, `Menubar`, `Navigation Menu`, `Pagination`, `Popover`, `Progress`, `Scroll Area`, `Separator`, `Skeleton`, `Tooltip`, `Typography`.
- **Typography — единственная точка входа для текста.** Прямые размеры шрифтов в компонентах запрещены.
- **Этот Guide — контракт.** Если код и Guide расходятся, это считается багом. Нельзя оставлять расхождение "на потом".

### Непереговорные правила
- Все числа интерфейса живут в общей системе: `--space-*`, `--type-*`, `--width-*`, `--radius-*`, `--opacity-*`, avatar/flag size-scale.
- Если блок "ощущается неправильным", сначала меняется общий токен или выбирается другой уже существующий токен. Локальная арифметика вида `calc(var(--space-lg) * 0.9)` для одного блока запрещена.
- Локальные токены под один экран, страницу или блок запрещены. Нельзя заводить `hero-gap`, `home-inset`, `episode-width`, `about-padding` и любые их аналоги.
- Семантические CSS-классы допустимы, но их численные значения должны ссылаться только на общие токены. Семантический класс не даёт права вводить новую локальную шкалу.
- Нельзя смешивать несколько конкурирующих способов задать один и тот же ритм. Если вертикальный ритм контейнера задаётся через `gap`, дочерние элементы не должны дублировать тот же ритм через `margin-top`.
- Не существует понятия "особого случая для одного блока". Если правка обсуждается на примере hero, navbar, episode, character или любой другой конкретной зоны, решение всё равно должно оформляться через общую систему.
- Любая визуальная правка обязана завершаться синхронизацией кода и документации в том же изменении. Временное "сейчас поправим код, а Guide потом" запрещено.
- Любое изменение токенов считается системным изменением. После него обязательно обновляется этот Guide.

### Запрещённые приёмы
- Tailwind-spacing классы (`gap-*`, `p-*`, `px-*`, `py-*`, `m-*`, `mt-*` и т.д.) как источник дизайн-ритма.
- Raw `font-size`, `gap`, `padding`, `margin` в feature-коде и design-system коде, если то же можно выразить токеном.
- Локальные `clamp()` и локальные пропорции для одного конкретного блока.
- "Подкрутить только здесь" ради визуального совпадения без изменения общей шкалы.
- Держать в коде одновременно старую и новую систему размеров.
- Оставлять в коде старый слой после миграции на новый (`legacy widths`, `temporary spacing`, `потом удалим`).
- Вводить новые семантические классы, которые маскируют локальные числа под системный слой.

### Политика работы с shadcn
- `shadcn` используется как исходный код компонентов в проекте, а не как "неизменяемая библиотека".
- Базовое поведение (`a11y`, keyboard, overlays, focus management) берётся из Radix/shadcn.
- Визуальный слой всегда приводится к токенам Aeria (`--bg`, `--text`, `--accent`, `--space-*`, `--type-*`, `--opacity-*`).
- "Сырые" shadcn-компоненты в фичах не используются напрямую: только через наши компоненты из `components/ui`.
- Если поведение уникальное для Aeria и не совпадает с shadcn-паттерном, компонент делается кастомным.
- **Avatar** остаётся единым базовым примитивом shadcn/Radix с size-variants `xs/sm/md/lg` (дубликаты компонента не создаются).
- Числовая шкала аватаров: `xs = 32px`, `sm = 75px`, `md = 128px`, `lg = 256px`.
- Для привязки к сущности используется отдельная обёртка `EntityAvatar` (всегда ссылка на связанную запись).
- Для стран используется отдельный примитив `Flag` (только `flag_colors`, без click-behavior).
- Карточные подложки/обводки не являются стилем по умолчанию: запрещено оборачивать ими каждый блок страницы "для красоты".
- Карточный стиль допускается только там, где он реально выражает отдельный модуль/сущность или интерактивный фокусный блок.

### Кастомный блок: Avatar Conveyor
- Это отдельный кастомный компонент (не стандартный shadcn): горизонтальный конвейер аватарок.
- Поведение:
  - автодвижение в одну сторону;
  - можно "подёргать" (drag/scroll) вручную;
  - клик по аватарке открывает связанную запись.
- Набор элементов фиксированный и задаётся заранее автором (без рандома в runtime).
- Каждая аватарка обязана ссылаться на конкретную сущность.
- Для персонажей аватарка обязательна; для неперсонажей (локации/объекты/иное) опциональна.
- Источник изображения определяется самой сущностью (персонаж/локация/атлас-элемент знает свой image asset).
- Компонент использует общий size-scale аватаров `xs/sm/md/lg` по дизайн-токенам.

### Кастомный блок: Country Media Carousel
- На главной странице это отдельный блок между `Обо мне` и `Лентой`.
- Основа: `Carousel/CarouselContent/CarouselItem` (shadcn + Embla) без форка `ui/carousel`.
- Данные v1 фиксированные и хранятся внутри компонента:
  - 6 hardcoded слайдов;
  - mixed media (`image` и `video`);
  - временные локальные media-файлы из `public/assets/media/countries`.
- Флаг на слайде рендерится через `Flag` из hardcoded `country`-объекта (`id/slug/title_ru/flag_colors`), без API-запросов.
- Геометрия и ритм:
  - ширина блока `width-wide`;
  - высота через `AspectRatio` с ratio `2240 / 1080`;
  - overlay-контент: слева, вертикально по центру;
  - внутренний inset overlay: `--space-xl`;
  - расстояния `title -> flag -> description`: `--space-md`.
- Переключатели под медиа:
  - отступ до точки-блока: `--space-md`;
  - inactive dot: вторичный тон;
  - active dot: pill в 2x шире базовой точки, основного тона, с плавной анимацией.
- Поведение:
  - autoplay каждые 7 секунд;
  - autoplay останавливается при `hover/focus/drag` и возобновляется после завершения взаимодействия.

### Кастомный блок: Услышано (цитаты из мира)
- На главной блок размещается сразу после `Country Media Carousel`.
- Разделитель между каруселью и блоком цитат: `SectionBreak` в варианте `stars` (без дополнительной линии).
- Данные:
  - источник — таблица `world_quotes` в PostgreSQL;
  - в `GET /api/home` приходит стартовая `world_quote`;
  - `GET /api/home/world-quote/random` отдаёт случайную цитату для действия "Подслушать ещё".
- Формат:
  - цитата рендерится в кавычках с пробелами: `« {текст} »`;
  - подпись всегда рендерится как `— {source} —`.
- Ритм и типографика:
  - цитата использует максимальный размер из типографической шкалы (`--type-extra`) через `Typography`;
  - подпись и текст кнопки — стандартный `body`;
  - отступ `цитата -> подпись`: `--space-sm`;
  - отступ `подпись -> кнопка`: `--space-lg`.

### Межсекционный разделитель: SectionBreak
- Для разрывов между контентными секциями используется единый компонент `SectionBreak`.
- Варианты:
  - `line` — классическая линия-разделитель;
  - `stars` — декоративный кластер из 3 звёзд.
- Оба варианта обязаны использовать один и тот же вертикальный ритм: `margin-block: var(--space-xl)`.
- `stars` может заменять линию в конкретных переходах между секциями, но не дополняется второй линией до/после себя.
- В `stars` звёзды декоративные (`aria-hidden`), не кликабельны и вращаются от прокрутки страницы.

### Примитив: Flag
- `Flag` — отдельный компонент, который принимает объект страны.
- Рендерится только если у страны есть `flag_colors`.
- Флаг представляет собой прямоугольную полоску из `N` равных сегментов (без зазоров), где `N = flag_colors.length`.
- В v1 `Flag` не кликабелен.
- В v1 источник флага: только `flag_colors`.

### Сетки и скейлы (ratio-based)
Исходные значения из Figma — это **соотношения**, а не абсолюты. На web они переводятся в `clamp()` между 360–1920.

**Брейкпоинты**
- 360 / 768 / 1280 / 1440 / 1920

**Типографика (4 размера)**
- `extra` / `medium` / `normal` / `small`
- Соотношение из Figma: 96 / 64 / 36 / 24
- В коде: `--type-extra`, `--type-medium`, `--type-normal`, `--type-small` через короткие fluid-формулы `clamp(min-rem, preferred-rem+vw, max-rem)`.
- Web-шкала калибруется оптически относительно Figma, а не переносится по номинальным px один-в-один: браузерный рендер и реальные шрифты могут визуально выглядеть крупнее макета при одинаковом числе px.
- Текущие desktop-ориентиры для web:
  - `extra`: около `95px`
  - `medium`: около `68px`
  - `normal`: около `39px`
  - `small`: около `28px`
- Используются только эти размеры, без локальных исключений.
- Все тексты оформляются через `Typography` и роли (`ui`/`heading`/`body`).
- Основной контентный `body` использует размер `normal`; меньше него по умолчанию допускается только `ui`.
- Если нужен UI-шрифт крупнее обычного UI, меняется роль шрифта (`fontRole="ui"`) при существующем `Typography variant`, а не создаётся отдельный локальный размер.
- Текстовые размеры не масштабируются "на глаз" внутри конкретного блока. Масштаб меняется только через общую типографическую шкалу.

**Отступы (5 размеров)**
- Шкала: `xs` / `sm` / `md` / `lg` / `xl`
- Целевой визуальный ориентир на экране `2560x1440`: примерно `10 / 20 / 40 / 80 / 160`.
- Шаг между соседними токенами сохраняется как `x2`.
- В коде это задано как короткие `clamp(min-rem, preferred-rem+vw, max-rem)` без прямой привязки к `vh`:
  - `--space-xs: 0.3125rem ... 0.625rem`
  - `--space-sm: 0.625rem ... 1.25rem`
  - `--space-md: 1.25rem ... 2.5rem`
  - `--space-lg: 2.5rem ... 5rem`
  - `--space-xl: 5rem ... 10rem`
- Между логически связанными блоками — ближе, между несвязанными — дальше.
- Для микро-отступов используется базовая шкала `--space-1 ... --space-10`; для layout и секций — только `--space-xs ... --space-xl`.
- Отступы внутри блока должны читаться как одна система. Правильный паттерн: контейнер владеет ритмом через `gap`, элементы внутри не дублируют тот же ритм через локальные margin.
- Если нужный отступ не укладывается в текущую шкалу, сначала пересматривается глобальная шкала. Локальный множитель для одного блока запрещён.

**Скругления (3 размера)**
- Шкала: `sm` / `md` / `lg`
- В коде: `--radius-sm = 20px`, `--radius-md = 30px`, `--radius-lg = 50px`
- Блоки и интерактивные поверхности используют только эти токены; локальные `border-radius`-значения для отдельных экранов не допускаются.

**Непрозрачности (только для текста)**
- `100 / 64 / 36 / 8` → `--opacity-100/64/36/8`
- 100 — основной текст, 64 — вторичный, 36 — третичный, 8 — разделители/подложки.

**Колонки**
- `narrow` / `medium` / `wide` в Figma: 880 / 1400 / 2240
- В коде: `--width-narrow/medium/wide` через `clamp()` с `rem`-границами и `vw` как fluid-серединой.
- Эти три ширины используются и на desktop, и на mobile; отдельного мобильного container-token нет.
- Любой блок сайта обязан жить в одной из этих трёх колонок. Нельзя "чуть-чуть поджать" или "чуть-чуть расширить" конкретный блок локальным значением.

### Аудит после визуальной правки
Любая правка дизайна проходит один и тот же порядок:
1. Определить, какая система сломана: `type`, `space`, `width`, avatar scale, flag scale, opacity.
2. Исправить именно эту систему, а не конкретный блок.
3. Поиском по репозиторию убедиться, что не появился новый локальный слой: raw-числа, локальные `clamp()`, page-specific token, дублирующий utility-класс.
4. Сверить Guide и код. Если токены или правила поменялись, Guide обновляется сразу.
5. Проверить как минимум главную, эпизоды, персонажей, атлас и навбар, чтобы убедиться, что системная правка не поломала соседние экраны.

### Если код и Guide расходятся
- Источник истины не может быть "и там, и там по чуть-чуть". После обнаружения расхождения задача не считается завершённой.
- Если правило устарело, сначала переписывается Guide и в том же изменении синхронизируется код.
- Если правило актуально, а код ему не соответствует, исправляется код. Временных исключений не допускается.
- Любое обсуждение визуальной проблемы должно заканчиваться ссылкой на систему, которая за неё отвечает. Ответ "подкрутим этот один блок" считается неправильным по умолчанию.

**Шум**
- Gaussian noise на весь фон.
- По умолчанию **выключен** (`--noise-opacity: 0`), включается через настройки.
- Шум — глобальный слой, не применяется локально на карточки.

### Палитра тем
Каждая тема имеет **фон**, **текст**, **акцент**. Другие цвета выводятся только из этих трёх через смешивание/opacity.

**Paper**
- Light: фон `#FEFDFB`, текст `#2A2A2A`, акцент `#FFB44B`
- Dark: фон `#121212`, текст `#F1EEE9`, акцент `#F2CA86`

**Coral**
- Light: фон `#FFF6F4`, текст `#392D31`, акцент `#F95C4B`
- Dark: фон `#271C1D`, текст `#FFE6E1`, акцент `#FF7D6F`

**Stone**
- Light: фон `#E4DED2`, текст `#2B2621`, акцент `#8D6B44`
- Dark: фон `#1D1A18`, текст `#E8DFD2`, акцент `#D0A878`

**Amoled**
- Dark (основной): фон `#000000`, текст `#F1EEE9`, акцент `#72A5FF`
- Light: фон `#F2F2F0`, текст `#141414`, акцент `#6EA8FF`

**Исключение**
- Цвет серии эпизодов может быть вне палитры темы и отображается как самостоятельный акцент.

### Mobile-режим (правила)
- Основной размер body не опускается ниже `--type-normal`.
- Заголовки масштабируются только через `--type-*` (никаких ручных размеров).
- Длина строки для чтения: 45–75 символов.
- Вертикальный ритм:
  - связанные блоки: `--space-sm` / `--space-md`
  - независимые блоки: `--space-lg`
  - глобальные разрывы секций: `--space-xl`
- На mobile продолжают работать те же `width-narrow / width-medium / width-wide`; из-за общего `gutter` они естественно схлопываются в одну безопасную ширину.
- Приоритет: контекст + действие, без перегруза (progressive disclosure).

## Границы ответственности
- **Контент** редактируется только в Markdown, API и UI — читатели.
- **Схема БД** только через миграции, без ручных правок.
- **UI** развивается через дизайн-систему, без хаотичных вариантов.
- **Серверные данные во фронтенде** хранятся через TanStack Query, а не в постоянном глобальном store.
- **DTO публичного API** не раскрывают служебные поля импорта (`source_path`, `content_hash`, `archived_at`).

## Порядок изменения дизайна
1. Сначала определить, какая из существующих систем отвечает за проблему: `type`, `space`, `width`, avatar scale, flag scale, opacity.
2. Если проблема системная, меняется соответствующий общий токен.
3. Если проблема не решается существующей шкалой, вводится новый токен только как часть общей шкалы, а не как исключение под один блок.
4. После изменения токена проверяются все ключевые страницы, а не только тот блок, который спровоцировал правку.
5. Guide обновляется в том же изменении, что и код.
