# Shadcn Policy (Aeria)

Цель: использовать shadcn/Radix как слой поведения и a11y, а визуал и контракты держать в дизайн-системе Aeria.

## 1) Базовый принцип
- Shadcn в проекте не "чёрный ящик".
- Разрешено адаптировать компонент под токены и UX-контракты Aeria.

## 2) Обязательные правила
1. Цвета, размеры, отступы, радиусы, opacity — только через системные токены.
2. В feature-коде использовать только наши компоненты из `components/ui` и `components/entities`.
3. Типографика только через `Typography` (`ui|heading|body`).
4. Состояния `hover/focus/active/disabled` фиксировать в компонентах, не в page-local CSS.
5. Для underline-состояний использовать только общий слой `accent-underline*`.

## 3) Запрещено
1. Оставлять дефолтный shadcn-визуал как финальный.
2. Делать одноразовые визуальные форки под конкретную страницу.
3. Вводить локальные размеры вне токенов.
4. Дублировать примитивы, если задача решается variant/обёрткой.

## 4) Avatar policy
1. Базовый `Avatar` один (shadcn/Radix).
2. Размеры только через variants: `xs | sm | md | lg`.
3. Дубликаты Avatar-компонента запрещены.
4. Сущностная обёртка — `EntityAvatar`:
- знает `entityType` и `entitySlug`,
- строит ссылку по типу сущности,
- используется там, где нужен переход на detail.

## 5) Flag policy
1. `Flag` — отдельный примитив, не часть Avatar.
2. Источник в v1: только `flag_colors` страны.
3. Рендер: `N` равных сегментов без зазоров (`N = flag_colors.length`).
4. Если `flag_colors` пустой/нет — компонент не рендерится.
5. В v1 `Flag` не кликабелен.

## 6) Кастомные компоненты
Кастомный компонент создаётся, если поведение не покрывается shadcn-паттерном без ущерба UX.

Текущие кастомные слои:
- `Navbar`
- `EntityAvatar`
- `Flag`
- `Avatar Conveyor`
