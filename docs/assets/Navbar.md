# UX/UI settings — Aeria

Документ фиксирует базовые UX-ограничения для проектирования интерфейса Aeria.

## Reference usage policy

- Референс-скриншоты используются для считывания **mood / ритма / плотности** интерфейса.
- Референсы **не являются пиксельным шаблоном** и не копируются «как есть».
- Допускается переносить:
  - композиционные принципы;
  - визуальный темп (чередование спокойных и насыщенных зон);
  - логику иерархии контента.
- Не допускается:
  - буквальное повторение верстки референса 1:1;
  - копирование фирменных визуальных паттернов другого продукта.

## Page roles

Каждая страница должна иметь одну ведущую роль. Роли не смешивать внутри одного экрана.

### Главная

- Назначение: вход в мир и текущий контекст чтения.
- Содержит только ключевые CTA и краткие точки входа.
- Не превращается в энциклопедию или полный листинг контента.

### Листинг глав

- Назначение: последовательная навигация по главам.
- Содержит структуру, фильтры/сортировку (при необходимости), статус прочтения.
- Не перегружать энциклопедическими карточками персонажей и мира.

### Персонажи

- Назначение: энциклопедия персонажей и связи между ними.
- Содержит карточки, биографии, связи, теги и связанные материалы.
- Не смешивать с полноценным листингом глав на одном экране.

### Мир

- Назначение: справочник по устройству мира (локации, фракции, сущности, термины).
- Содержит структурированные знания и перекрестные ссылки.
- Не подменяет главную и не дублирует страницу персонажей.

### Мир — дизайн-вектор (зафиксировано 27 февраля 2026)

- Визуальный тон: **premium editorial**, дерзкий и контрастный, без «безопасного» нейтрального UI.
- Композиция: асимметрия, крупные поля воздуха, явная вертикальная ритмика, акцент на длинные линии чтения.
- Типографика: крупные заголовки и выразительные перепады масштаба, интерфейсная типографика вторична.
- Контраст: смелые переходы светлого/темного слоя внутри одной страницы, но без цветового шума и кислотных акцентов.
- Навигация по контенту: приоритет исследовательского сценария (поиск, фасеты, связи), а не декоративных карточек.
- Структура листинга: избегать «карточного супа», предпочтение линейному editorial-flow с сильной иерархией.
- Detail-страница: hero + фактологическая полоса + основное чтение + вторичный контекст/связи; каждый блок должен иметь явную роль.
- Motion: минимальный, точный, без «игрушечного» поведения; анимация подчеркивает иерархию, не создает шум.

Нельзя для раздела «Мир»:

- Возвращаться к безопасному шаблону «типовой grid из одинаковых карточек».
- Делать низкоконтрастный «туманный» интерфейс без четких смысловых опор.
- Смешивать много декоративных приемов одновременно (градиенты, тени, шум, анимации) без функциональной причины.
- Сводить страницу к голому CRUD-представлению данных без редакционного ритма.

## Progressive disclosure rules

- По умолчанию показывается только минимум для принятия следующего действия.
- Детализация раскрывается по требованию пользователя (expand, «читать далее», переход в детальную карточку).
- На первом экране приоритет: контекст + действие, а не полный объем данных.
- Глубокие связи и второстепенные атрибуты уводятся на второй/третий уровень.
- Любой блок, который невозможно быстро просканировать, должен быть декомпозирован.

## PR checklist (UX scope)

- [ ] Не смешаны роли страниц (главная, листинг глав, персонажи, мир).
- [ ] Есть отдельные маршруты для списка глав и энциклопедических разделов.
- [ ] На главной нет перегруза контентом, не нарушен принцип progressive disclosure.

## Navbar — возможности и тексты

Навбар является главным системным слоем. Он должен одинаково понятно работать на desktop и mobile и не ломать контекст чтения.

### 1) Основная навигация по разделам

- Разделы: `Главная`, `Эпизоды`, `Персонажи`, `Атлас`.
- Клик по разделу ведет на корень раздела.
- Активный раздел визуально отмечен.
- На mobile разделы дублируются в нижней панели для постоянного доступа.

### 2) Breadcrumb-режим внутри разделов

- Если пользователь находится внутри карточки/детальной страницы, вместо списка разделов показывается путь:
  - `<Раздел> > <Текущая страница>`
- Клик по названию раздела в хлебных крошках возвращает к корню раздела.
- Кнопка `Назад`:
  - если в истории есть шаги, делает переход назад по истории;
  - если истории нет, возвращает на корневую страницу текущего раздела.

### 3) Контекст чтения главы

- При чтении главы в навбаре показывается прогресс:
  - формат: `<текущая глава>/<всего в серии>` (например, `3/12`);
  - подсказка при наведении: название серии.
- В верхней границе навбара показывается полоска прогресса прокрутки текущей главы.

### 4) Быстрые системные действия

- `День / ночь` — переключает светлый и темный режим.
- `Aa` — открывает панель шрифтов:
  - Заголовки;
  - Текст эпизодов;
  - Элементы UI.
- `...` — открывает панель настроек:
  - выбор темы (`Paper`, `Stone`, `Coral`, `Amoled`);
  - фон (`Шум вкл` / `Шум выкл`);
  - tap effect (`None` / `Ripple`).

Все изменения, сделанные через навбар (тема, режим, шрифты, шум, tap effect), **сохраняются в cookies сайта** и восстанавливаются при следующем визите.

### 5) Глобальный поиск по всему сайту (обязательный модуль)

Глобальный поиск должен быть встроен в навбар и искать одновременно по всем основным сущностям:

- Эпизоды;
- Персонажи;
- Атлас;
- Серии эпизодов;
- Страны и локации.

Требования к UX:

- Поле доступно с любой страницы.
- Клик по иконке поиска открывает **inline-строку под навбаром** (не модальное окно).
- Горячая клавиша открытия: `Ctrl/Cmd + K`.
- Поиск работает без перезагрузки страницы (SPA-переходы).
- Результаты поиска выводятся **плоским списком без карточных подложек, рамок и теней**.
- Результаты сгруппированы по типам (`Эпизоды`, `Персонажи`, `Атлас`, `Серии`, `Страны/Локации`).
- У каждого результата: заголовок, короткий вторичный текст, тип сущности, ссылка.
- Поддержка синонимов/альтернативных написаний и опечаток (минимум мягкий fuzzy-поиск).
- Пустой запрос показывает только последние открытые записи (recent); если recent нет — строка остается без текстовой заглушки.
- По `Enter`:
  - при выбранном результате — переход на него;
  - без выбранного результата — переход на страницу всех результатов.

Текстовые состояния:

- Placeholder поля: `Поиск по главам, персонажам и миру...`
- Пусто (для непустого запроса): `Ничего не найдено. Попробуйте другое слово.`
- Ошибка: `Поиск временно недоступен. Попробуйте позже.`
- Загрузка: `Ищем...`

### 6) Правила стабильности навбара

- Навбар не должен «прыгать» по высоте между страницами.
- Переключение режимов (меню/хлебные крошки/поиск) должно сохранять визуальный ритм.
- Все элементы навбара доступны с клавиатуры.
- На mobile важные действия не скрываются глубже одного тапа.
- В desktop-правой группе используется один вертикальный разделитель: между блоком `Поиск (+индикатор главы)` и блоком `День/ночь + Aa + ...`.
- Меню `Aa` и `...` открываются в одной фиксированной точке под навбаром у правой границы контентной области.

### 7) Единые underline-правила (обязательно)

- Подчёркивания в navbar не реализуются локальными дубликатами.
- Разрешены только общие классы:
  - `accent-underline` (всегда видимое),
  - `accent-underline-click` (активация по состоянию),
  - `accent-underline-hover` (активация по hover/focus-visible).
- Для click-состояния источником истины могут быть:
  - `.accent-underline-active`,
  - `data-underline-active="true"`,
  - `aria-current="page"`/`aria-expanded="true"`.
- Элементы `Главная/Эпизоды/Персонажи/Атлас`, кнопки `Поиск`, `Aa`, `...` обязаны жить на этом общем underline-слое.
